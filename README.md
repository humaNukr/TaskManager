# KMA TaskManager

## Опис проєкту
Система управління завданнями, побудована за принципами багатошарової архітектури. Реалізує повний цикл відображення проєктів та завдань — від шару даних до графічного інтерфейсу.

## Структура рішення
Проєкт розділений на декілька бібліотек для дотримання принципу **Single Responsibility**:

1. **Common**: Містить переліки (`Enums`), спільні для всіх шарів.
2. **DataModels**: Класи для сирого зберігання даних. Не містять обчислювальних полів та логіки відображення.
3. **UIModels**: Моделі для представлення даних у користувацькому інтерфейсі. Містять обчислювані властивості (прогрес, статус прострочення) та логіку форматування. Включає моделі для перегляду (`ProjectUIModel`, `TaskUIModel`).
4. **CreateModels**: Моделі для створення нових сутностей (`ProjectCreateModel`, `TaskCreateModel`). Містять лише вхідні дані від користувача, без обчислювальних властивостей.
5. **Storage**: Визначає інтерфейс `IStorageContext` та реалізацію `InMemoryStorageContext` — сховище даних у пам'яті з початковим набором тестових даних.
6. **Services**: Шар бізнес-логіки. Містить:
   - `Mappers`: Логіка перетворення даних між шарами (Data ↔ UI).
   - `TaskService` / `ProjectService`: Сервіси для фільтрації та отримання даних.
7. **ConsoleApp**: Консольний клієнтський додаток для взаємодії з користувачем через термінал.
8. **Maui**: Графічний застосунок на базі .NET MAUI. Реалізує навігацію між сторінками та відображення даних через Data Binding.
9. **Tests**: Юніт-тести для сервісів, маперів та UI-моделей (xUnit + Moq).

## Навігація у MAUI-застосунку
Застосунок використовує `Shell`-навігацію і містить три сторінки:

- **MainPage** — відображає список усіх проєктів з назвою та відсотком прогресу.
- **ProjectDetails** — детальна інформація про проєкт: назва, тип, опис, прогрес-бар, список завдань.
- **TaskDetails** — детальна інформація про завдання: назва, статус, пріоритет, дедлайн, прострочення, опис.

Навігація відбувається через `Shell.Current.GoToAsync` з передачею обраного об'єкта як `QueryProperty`. Нові вікна не створюються — всі переходи відбуваються в межах одного вікна застосунку.

## Ключові принципи
- **Інкапсуляція**: Доступ до сховища має тільки шар сервісів через інтерфейс `IStorageContext`.
- **Dependency Inversion**: Сервіси залежать від абстракцій (`IStorageContext`, `IProjectMapper`, `ITaskMapper`), а не від конкретних реалізацій.
- **Ізоляція рівнів**: Об'єкти `DataModel` не містять колекцій дочірніх об'єктів — зв'язок реалізовано через ідентифікатори (`Guid`).
- **Lazy Loading**: Завдання завантажуються лише після переходу на сторінку конкретного проєкту.
- **Незмінність ідентифікаторів**: Поля `Id` та `ProjectId` у моделях даних доступні лише для читання — їх неможливо змінити після створення об'єкта.
